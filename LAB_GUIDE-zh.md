# 第2周实验指南 — 上下文工程：记忆、CLAUDE.md 与 RAG

**时间预算：** 约 35–40 分钟（在 90 分钟现场课程内完成）。

**前提条件：** 能够通过 Claude Code 完成多步骤任务（第1周）。今天的幻灯片讲解了扎根的三个层次——CLAUDE.md、记忆、RAG——以及什么是上下文债务。如果上述术语有任何不熟悉的，请先回顾幻灯片再继续。

---

## 1. 设置

1. 克隆本仓库并在终端中打开。
2. 创建虚拟环境、安装依赖，并确认基准测试通过：
   ```
   python3 -m venv .venv
   source .venv/bin/activate
   pip install -r requirements.txt
   pytest
   ```
   全部 4 个测试应通过。这就是 **Ledger**——一个小型的、可正常运行的订单/计费服务。这里没有任何需要修复的东西，也不需要重构。今天的问题不在代码——而在代码*周围*那些没有被记录下来的内容。
3. 暂时不要打开 `CHANGELOG.md`。留到第1步再看。

---

## 2. 快速参考卡

幻灯片刚讲过这些内容——列在这里，是为了让你在练习时不需要切回去翻找。

| 层次 | 涵盖内容 | 生命周期 |
|---|---|---|
| CLAUDE.md | 约定、架构说明、做/不做 | 静态——每次会话相同 |
| 记忆 | 已做的决策、积累的偏好 | 随时间积累 |
| RAG | 大型或变化中的文档/代码库 | 按需检索 |

**判断标准：** 它变化的频率如何？体量有多大？稳定 + 小 → CLAUDE.md。持久的决策 → 记忆。体量大 + 持续变化 → RAG。

## 3. 练习

### 第1步——冷启动审计 Ledger

打开 `ledger.py`。不要先运行 Claude Code，先自己回答：**要安全地修改这段代码，你需要了解什么？** 具体来说：

- `PaymentClient.charge()` 的 `idempotency_key` 参数实际防范的是什么——如果调用方每次重试时都生成一个*新的*键，而不是复用同一个键，会发生什么？
- 为什么 `refund_order()` 对部分金额会抛出 `NotImplementedError`？这是 bug、缺失的功能，还是一个决策？

现在查看 `docs/architecture.md`。它回答了上述任何一个问题吗？再查看 `CHANGELOG.md`——这个文件就是幻灯片里"那条六个月前的 Slack 消息"的替代品。在真实代码库中，这类决策往往就是这样——埋在 changelog、已关闭的 PR 描述或旧 issue 帖子里，而不是聊天记录里。

**检查点：** 你刚才需要费力挖掘的内容——那就是真实的上下文债务。现在把同样的问题问 Claude Code（*"我需要了解什么才能安全修改 `ledger.py`？"*）——在做其他任何事之前先问——然后把它的回答与你自己找到的内容对比。

### 第2步——为 Ledger 编写 CLAUDE.md

向 Claude Code 发出指令：

> 通读这个仓库（包括 CHANGELOG.md），为它编写一份 CLAUDE.md。内容覆盖：项目概述、约定、做/不做、架构说明以及测试命令。保持简洁——如果一条规则需要一段话来说明理由，把那段话放进对应的源文件或文档里，而不是放进 CLAUDE.md。

**检查点：** 读一遍它生成的内容。幂等键约定是否出现在"做/不做"里？部分退款限制是否作为"不做"项出现，并附上一行指向*原因*的说明（这是一个被推迟的产品决策，不是 bug），而不是把完整背景都塞进来？

### 第3步——捕获那个在新会话中最难被发现的决策

指令 Claude Code 保存一条记忆条目——项目层级，与幻灯片中的分类相同——记录部分退款的决策：从 v1 起推迟，需要产品团队 sign-off，才能开始任何工程实现。一句话就够了。

**检查点：** 开启一个**新的** Claude Code 会话（或清除上下文），然后问：*"我可以安全地给 Ledger 添加部分退款功能吗？"*——不做任何额外说明。它会主动提及那条记忆条目吗？

### 第4步——前后对比

用同一个问题——*"我可以安全地给 Ledger 添加部分退款功能吗？"*——想象一下在没有 CLAUDE.md、没有记忆的情况下提问。你可以真实测试这一点：临时把 `CLAUDE.md` 改个名移开，提问，然后还原，再提问。

- **之前：** 给出一个合理的实现草图，完全不知道这个功能是被刻意推迟的。
- **之后：** 提出被推迟的决策，而不是悄悄重新打开一个已经关闭的问题。

同样的模型、同样的问题、同样的代码——差别就是你刚才构建的一切。

## 4. 反思提示

与小组讨论（或独自作答时记录下来）：

1. 第1步中有什么让你感到意外——Claude Code 冷启动时答错了什么，或者你自己差点也漏掉了什么？
2. 你选择**不**放进 CLAUDE.md 的是什么，为什么？
3. 部分退款决策真的算"上下文债务"吗？还是说 `NotImplementedError` 加上一条 changelog 记录已经是合理的记录方式了？好到够用和真正缺失之间的界限在哪里？
4. 如果 Ledger 还有十个像退款这样的决策，记忆条目还能撑得住规模吗？还是说你会改用 RAG 来管理一个真正的决策日志？
